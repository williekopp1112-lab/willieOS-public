Paperclip vs Gary

A Competitive Analysis of Multi-Agent Architectures

Paperclip is a control plane for autonomous AI companies.
Gary is a deeply integrated AI Chief of Staff.

Both orchestrate multiple agents. Both excel at different things.
This deck identifies exactly what to steal.

March 26, 2026 • Willie Kopp

Executive Summary

Paperclip

Philosophy: Multi-company control plane with strict hierarchical delegation

Target Use Case: Running autonomous AI companies with CEO agents, formal org charts, and task ancestry chains

Core Strength: Governance, delegation, and multi-tenant architecture with budget controls and audit trails

Gary (WillieOS)

Philosophy: Single-CEO personal chief of staff with rich contextual memory

Target Use Case: Deep personal integration with calendar, email, relationships, and knowledge — orchestrating specialist agents when needed

Core Strength: Context richness, learning systems, real-world integrations, and proactive intelligence

What Paperclip Does Better

1. Goal Ancestry Chains

Every task traces back to the company's top-level mission. "If you can't explain why a task matters to the company goal, it shouldn't exist." This prevents work that doesn't ladder up to strategic objectives.

2. Agent Capabilities Discovery

Agents describe their capabilities in a structured format. Other agents query "who can help with X?" instead of hardcoded routing. Enables dynamic team composition and specialization discovery.

3. Atomic Task Checkout

Single-assignee model with checkout semantics prevents race conditions. Two agents can't accidentally work on the same task simultaneously. Critical for coordination at scale.

4. Formal Delegation Cascades

CEO → CTO/CMO/CFO → specialists. Each agent has a reports_to field. Enables proper responsibility chains and escalation paths in complex organizations.

5. Pluggable Memory Abstraction

Memory provider interface supports swapping backends (mem0, supermemory, Memori, etc.). Different agents can use different memory systems. Enables best-of-breed memory solutions per use case.

6. Budget Hard-Stop Auto-Pause

Monthly UTC budgets per agent with automatic pause when limit hit. Prevents runaway costs. Soft alerts before hard limits. Critical for production deployment confidence.

7. Immutable Audit Trails

Every action logged with tool-call tracing. Config changes are versioned with rollback. Ticket-based communication creates permanent records. Essential for compliance and debugging.

What Gary Already Does Better

1. Deep Personal Context

Willie Knowledge Graph tracks people, projects, priorities, blockers, goals, concerns, decisions, energy levels, family, sentiment, follow-up needs. Relationship intelligence with aliases and how_known fields. Paperclip has none of this.

2. Learning Engine

teach_gary for preferences, correction recording, confidence scoring, knowledge gap detection. Gary gets smarter over time. Paperclip agents are stateless between tasks.

3. Memory Diversity

Loops, notes, contacts, dispatch memories, settings, check-in data. Paperclip only has tasks + comments. Gary's memory is richer across more dimensions of human life and work.

4. Real Integrations

Google Calendar, Gmail, iMessage, Drive, GitHub Pages. Gary touches the real world. Paperclip agents only talk to each other. This is a fundamental architectural difference.

5. Dispatch Chains with Handoff Packets

Multi-step dependency chains auto-trigger on completion. get_handoff_packet assembles context from prior dispatches. Project tagging for cross-session recall. Continuity across complex workflows.

6. Proactive Intelligence

Daily check-ins, knowledge gap detection, self-assessment metrics, nightly performance trends, weekly improvement reports. Gary doesn't wait to be asked — he surfaces what matters.

7. Workflow Automation

Playbooks chain MCP tools (morning_autopilot, meeting_prep_full). Scheduled tasks with guardrails (debounce, cooldown, daily caps, concurrency limits). Email triage, stale loop sweep, proactive scan all automated.

Top 5 Improvements to Steal

P0 1. Goal Ancestry Chains Complexity: Medium

What: Every dispatch/loop must trace back to a strategic objective. Six-level hierarchy: Vision → OKR → Initiative → Project → Milestone → Task.

Why it matters: Prevents busywork. If Gary can't explain why a task ladders up to Willie's goals, it shouldn't exist. Forces ruthless prioritization.

How to adapt: Add `goal_id` foreign key to loops table. New `goals` table with parent_goal_id for hierarchy. UI shows goal path for every loop. Gary's system prompt enforces "no orphan tasks."

P0 2. Agent Capabilities Registry Complexity: Low

What: Agents describe capabilities in structured format. Other agents query "who can help with X?" Discovery-based routing replaces hardcoded keywords.

Why it matters: Current agentRegistry.js uses brittle keyword matching. As agent count grows, keyword collisions and coverage gaps multiply. Capabilities enable dynamic discovery.

How to adapt: Extend agentRegistry.js schema: add `capabilities: []` array (structured tags like {domain: "presentation", format: "pptx", actions: ["edit", "verticalize"]}). Add `query_agent_capabilities(query)` MCP tool. Gary uses semantic matching instead of keywords.

P1 3. Budget Tracking Per Dispatch Target Complexity: Medium

What: Monthly budgets per agent with soft alerts at 75% and hard auto-pause at 100%. Cost tracking per dispatch target (Claude Code vs Cowork vs background jobs).

Why it matters: Gary's cost tracking is global. Can't answer "how much does Cowork cost per month?" or "is this dispatch target worth the spend?" Need per-agent cost visibility for optimization.

How to adapt: Add `dispatch_target_budgets` table with monthly limits. Extend costTracker.js to track spend by target. Add `check_budget_before_dispatch()` gate. Notification at 75%, auto-pause at 100% with human override.

P1 4. Atomic Task Checkout Complexity: Low

What: Single-assignee model with checkout semantics. Before working on a loop/dispatch, agent must acquire exclusive lock. Prevents double-dispatch of same work.

Why it matters: Currently Gary can dispatch the same task twice if triggered by multiple events (email + calendar + proactive scan). Wastes money and creates conflicts.

How to adapt: Add `checked_out_by` and `checked_out_at` to gary_actions table. Before dispatch: `SELECT FOR UPDATE` to check availability. Release lock on completion or timeout (2 hours). UI shows locked tasks with owner.

P2 5. Formal Delegation Hierarchy Complexity: High

What: Agents have reports_to field creating org tree. CEO reviews VPs, VPs review specialists. Enables escalation paths and delegation cascades.

Why it matters: Long-term vision for Gary is managing an executive team (CTO, CMO, CFO). Need formal reporting structure for delegation. "CEO Gary delegates to CTO Gary who dispatches to Code."

How to adapt: Extend agentRegistry.js: add `reports_to` field. Create "VP-level" agents (CTO, CMO, CFO) that sit between CEO Gary and specialist agents. Routing logic: Gary → VP → specialist. Each VP has review cycles checking subordinate work.

Recommended Roadmap

Phase 1: Quick Wins 1-2 Weeks

  • Agent Capabilities Registry: Extend agentRegistry.js with capabilities array, add query_agent_capabilities MCP tool, update dispatch routing to use semantic matching
  • Atomic Task Checkout: Add checkout fields to gary_actions table, implement SELECT FOR UPDATE locking, add timeout release (2 hours), show locked tasks in UI
  • Basic Goal Tracking: Create goals table with parent_goal_id, add goal_id to loops, UI shows goal path, no enforcement yet

These slot into existing architecture without breaking changes. Low risk, immediate value.

Phase 2: Medium-Term 1 Month

  • Budget Tracking Per Target: Create dispatch_target_budgets table, extend costTracker.js for per-target tracking, implement soft alerts (75%) and hard limits (100%), add budget dashboard in Settings
  • Goal Ancestry Enforcement: Gary's system prompt requires goal_id on all new loops, no orphan tasks allowed, weekly review surfaces goalless work, migration tool to backfill existing loops
  • Dispatch Chain Improvements: Formal handoff contract schema (what gets passed between steps), validation gates before fan-in, cost/context budgets per chain

Requires schema changes and new enforcement logic. Some behavior changes for users.

Phase 3: Long-Term 2-3 Months

  • Formal Delegation Hierarchy: Add reports_to to agentRegistry.js, create VP-level agents (CTO/CMO/CFO), implement delegation cascades (CEO → VP → specialist), VP review cycles checking subordinate work
  • Pluggable Memory Providers: Abstract memory interface layer, support multiple backends (current SQLite + optional Turso + optional vector DB), per-agent memory provider overrides, cost/latency observability
  • Heartbeat-Driven Agent Lifecycle: Beyond cron — agents wake on schedule, review their domain, report status to CEO Gary, surface proactive opportunities, adaptive scheduling based on workload

Fundamental architecture evolution. Enables true executive team model. High complexity, high payoff.

Key Takeaways

Paperclip and Gary solve different problems exceptionally well.
Paperclip = control plane for AI companies.
Gary = deeply integrated AI chief of staff.

Steal from Paperclip

  • Goal ancestry chains (no orphan tasks)
  • Agent capabilities discovery (semantic routing)
  • Atomic task checkout (prevent double-work)
  • Budget controls per dispatch target
  • Formal delegation hierarchy (long-term)

Keep Gary's Strengths

  • Deep personal context (knowledge graph)
  • Learning engine (preferences, corrections)
  • Real-world integrations (Calendar, Gmail, etc.)
  • Proactive intelligence (gap detection, self-assessment)
  • Workflow automation (playbooks, guardrails)
Next Step: Start with Phase 1 quick wins (capabilities registry + atomic checkout + basic goal tracking). Ship in 1-2 weeks. Validate before Phase 2.
